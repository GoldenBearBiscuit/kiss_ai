KISS Sorcar — Frontend ↔ Backend task flow

Precise, step-by-step trace of what happens between the chat UI and the Python agent when a user submits a task — covering both viewers (VS Code extension webview remote browser (web app)) and the single shared kiss-web daemon they both talk to.

One‑sentence summary. Both the VS Code webview and a remote browser drive one and the same Python process (the kiss-web daemon) over a newline‑delimited JSON protocol — the extension over a Unix‑domain socket (~/.kiss/sorcar.sock, FS‑permission‑gated), the browser over WSS with password auth. A frontend submit becomes a backend run, which starts a background task thread that constructs a WorktreeSorcarAgent, calls agent.run(...), and emits events through WebPrinter.broadcast(...) back out to every connected viewer that subscribes to the task’s tab id.

  1. printer.print(system_prompt, type="system_prompt") and type="prompt" for the user message → broadcast a prompt / system_prompt event;
  2. stream from the model via the active Model subclass, calling printer.token_callback(token) for every token → coalesces into text_delta / thinking_delta events;
  3. when the model returns a tool call, broadcast {type:'tool_call', name, tool_input, tabId};
  4. execute the tool (_execute_tool) — Bash output is streamed live as system_output events, others append their stdout / return value when done — then broadcast {type:'tool_result', tool_name, content, is_error?, tabId} (suppressed only for finish);
  5. loop until finish is called or max_steps is hit. finish ends the loop; BaseBrowserPrinter._broadcast_result emits the final {type:'result', text, success, is_continue, summary, total_tokens, cost, step_count}.

3. Why one process, two transports

Under the single‑daemon architecture (USER_PREFS phases 1–5), the VS Code extension no longer spawns Python subprocesses. There is exactly one Python process — kiss-web, started by a macOS LaunchAgent or Linux systemd unit — and it owns one VSCodeServer instance and one WebPrinter: 

RemoteAccessServer.__init__():
    self._printer       = WebPrinter()              # owns the transport
    self._vscode_server = VSCodeServer(printer=self._printer)
    # one asyncio server bound to ('0.0.0.0', 8787)   ← WSS  (browser)
    # one asyncio.start_unix_server('~/.kiss/sorcar.sock', mode 0o600)  ← UDS (extension)

VS Code extension transport — UDS

  • Client: AgentClient — one persistent net.createConnection({path:'~/.kiss/sorcar.sock'}) per VS Code window, multiplexing every chat tab.
  • Wire format: newline‑delimited JSON (identical to WSS).
  • Auth: none — POSIX file permissions on the socket (mode 0o600) gate access to the owning UNIX user.
  • Reconnect: AgentClient._scheduleReconnect after socket 'close'; pending writes are queued in _pendingSends and flushed on next 'connect'.
  • Per‑connection state: RemoteAccessServer._uds_handler tracks every tabId seen on the connection; on disconnect it arms one deferred closeTab per tab (10 s grace).
  • Daemon mapping: the writer is registered with WebPrinter.add_uds_writer(writer); backend broadcasts fan out to _uds_writers in lockstep with _ws_clients.

Web app transport — WSS

  • Client: a vanilla browser WebSocket opened from the HTML page served by RemoteAccessServer._build_html. The HTML embeds a vscode‑compatible shim so main.js is oblivious to which transport it runs over.
  • Wire format: newline‑delimited JSON (one event per WS text frame).
  • Auth: _authenticate_ws challenges the client with {type:'auth_required'}; the browser replies {type:'auth', password:...}; constant‑time equality check against the remote_password in ~/.kiss/config.json; failures are rate‑limited per source IP.
  • Public reachability: an optional cloudflared tunnel (quick or named) is supervised by RemoteAccessServer._tunnel_loop; the public URL is published in ~/.kiss/remote-url.json and via a remote_url WS broadcast.
  • Per‑connection state: _ws_handler tracks every tabId seen on the connection. Because browsers cannot reliably emit closeTab before the WSS shuts down, disconnect arms one deferred closeTab per tab (_TAB_CLOSE_GRACE 10 s); a reconnect within the grace window cancels it.

Command translation

A few commands the webview emits target the VS Code host (filesystem picker, open editor, focus VS Code) and have no meaning over the WSS transport. They are listed in _VSCODE_ONLY_COMMANDS (web_server.py:452) and dropped on arrival. Two more are rewritten so the daemon sees a single canonical shape regardless of transport, in _translate_webview_command: 

userActionDone  → {type:'userAnswer', answer:'done', tabId}
resumeSession   →  rename field   id → chatId

submit is rewritten to run by both transports (extension: SorcarSidebarView._startTask; web: RemoteAccessServer._handle_submit) — see step 3 in §2.

4. Bidirectional mid‑task interactions

4.1 User answers an ask_user_question tool call

When the agent calls the ask_user_question tool, the printer broadcasts {type:'askUser', question, tabId} and the agent thread blocks on tab.user_answer_queue.get(timeout=...). In the webview, case 'askUser' (main.js:2467) renders an inline form with a Submit button. Clicking Submit posts back: 

{type:'userAnswer', answer:"...the user's text...", tabId}

Transport routes the line into the daemon → _CommandsMixin._cmd_user_answer → tab.user_answer_queue.put_nowait(answer) — the agent thread wakes up, the tool returns the string, the loop continues.

4.2 Stop button

Frontend: vscode.postMessage({type:'stop', tabId}) from the stop button (main.js:2786 / 3979).
Backend: _cmd_stop → tab.stop_event.set(). The next printer.print(...) from the agent thread raises KeyboardInterrupt via _check_stop; the task‑runner converts it into a task_stopped broadcast and falls through to the same shared cleanup as a normal completion.

4.3 Merge review (worktree mode)

When the task succeeds with useWorktree=true, the printer broadcasts merge_data (per‑file diff hunks). The extension intercepts that event and opens a native VS Code merge editor via MergeManager.openMerge instead of forwarding it to the webview (the webview only renders the in‑input merge toolbar with Prev / Next / Accept / Reject buttons). Over the web transport the very same event is augmented by WebPrinter._augment_merge_data with base_text + current_text, so the browser renders an inline diff inside chat. Toolbar clicks post {type:'mergeAction', action, tabId} back; the daemon’s _cmd_merge_action applies the hunk(s) and emits merge_nav / merge_ended.

5. Multi‑viewer fan‑out

Multiple clients can view the same running chat — a browser tab and a VS Code window simultaneously, or two browser tabs. This is driven by BaseBrowserPrinter._subscribers: 

_subscribers: dict[str, set[str]]    # source_tab_id → {viewer_tab_ids}

When a client opens a history row whose task is still running, VSCodeServer._replay_session finds the live _RunningAgentState (matched by agent.chat_id == chat_id), calls printer.subscribe_tab(source_tab_id, new_tab_id), and replays the in‑memory event buffer to the new viewer. From then on, every broadcast() emits the primary event under the source tab id, then one fan‑out copy per viewer ({**event, "tabId":viewer}) through the same downstream channel — but only the primary event is recorded / persisted, so the DB and replay buffer remain single‑sourced.

6. Persistence & history

Every display event passes through BaseBrowserPrinter._persist_event → _queue_chat_event(event, task_id=...). A single background thread (kiss-event-writer) batches up to 256 events / 20 ms and runs one executemany("INSERT INTO events ...") plus at most one UPDATE task_history SET has_events=1 per batch — producer latency stays flat under N=8 parallel agents (12 µs median). A _flush_chat_events() call drains the queue synchronously before any code path that takes a write lock on task_history (e.g. _save_task_result, _save_task_extra).

When the frontend clicks a row in the history sidebar, it sends {type:'resumeSession', id:chat_id, tabId}. The daemon rewrites id → chatId (see §3), and _replay_session either (a) re‑attaches to a still‑running task via _reattach_running_chat + subscriber fan‑out (broadcasts status running=true before task_events to match the fresh‑run order), or (b) reads the persisted events from sorcar.db and emits one {type:'task_events', events:[...], tabId} blob the frontend re‑renders with replayTaskEvents.

7. Lifecycle invariants

PropertyWhere enforcedWhy
tab_id == chat_id Frontend mints the UUID; _cmd_run adopts it as agent._chat_id; _replay_session uses the persisted chat_id as tabId for the new tab. Single‑key routing for events, persistence, and fan‑out.
status running=true before any event of a task _handle_submit and _startTask (optimistic); _run_task top (idempotent); _replay_session running branch (re‑attach). Frontend chevron state, timer, “Running …” label all key off it.
status running=false always emitted exactly once at task end _run_task’s outer finally. Stop button hides, timer stops, history row flips to “done”.
Closing a chat tab never interrupts a running agent _close_tab sets frontend_closed=True and returns when any of is_task_active / is_merging / task_thread.is_alive() hold; _dispose_if_closed tears the state down later from _run_task’s finally or _finish_merge’s tail. Users can close a tab and check back from history.
WS / UDS disconnect ≠ closeTab _schedule_tab_close defers a real closeTab by _TAB_CLOSE_GRACE (10 s); a reconnecting ready with the same tabId (or any restoredTabs entry) cancels the timer in _handle_ready. Reloads / drop‑and‑reconnect don’t kill running tasks.
One‑agent‑per‑tab at a time _cmd_run guard: if tab.task_thread and tab.task_thread.is_alive(): → broadcast error and bail. Avoids interleaved event streams under the same tab id.
